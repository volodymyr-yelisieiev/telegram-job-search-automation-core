# Release Gates

`ReleaseGateEvaluator` is the final aggregate gate before live automation. It consumes the release evidence ledger rather than hardcoded claims.

It blocks live/full automation unless all of these are true:

- provider readiness is green for every live auto-apply provider, using current live canary evidence plus explicit `PROVIDER_CONFIG_JSON` disable-switch coverage;
- production review-first/controlled submit readiness is backed by a live-capable provider module, not the local fixture provider runtime;
- the runtime is production-shaped: `NODE_ENV=production`, `STATE_BACKEND=postgres` with `DATABASE_URL`, `QUEUE_BACKEND=bullmq` with `REDIS_URL`, controlled/full automation mode, irreversible actions explicitly enabled, S3-compatible proof-artifact storage, live Telegram credentials plus guarded webhook secret, non-mock LLM transport configuration, and configured secret-store credentials when `SECRETS_BACKEND=local_encrypted_file`;
- live provider/Telegram credentials are configured;
- external secrets backend is configured;
- live canaries are passing;
- calendar integration is ready;
- dated 7-day soak has passed;
- provider submit proof and outbound dispatch proof exist;
- irreversible action gate is explicitly enabled when required by mode.

Telegram channel/source readiness is still reported, but it does not block the controlled auto-apply readiness check unless the provider is configured with `autoApply=true`. Live evidence and canary evidence still cover every expected provider/source.

The API surface is `GET /release-gates`. Local-safe fixture runs are expected to return `readyForLiveAutomation: false` until external evidence exists.

Replace every example value in `release-evidence.json` with real dated production evidence, every example value in `ga-signoff.json` with real product/engineering/operations/security decisions plus real `evidenceRefs` for the issue register, runbook drill report, residual-risk record, and post-GA maintenance plan, and generate a fresh `runtime-preflight.json` from the same production-shaped environment used for final audit. The templates include explicit example markers or local-only skipped-probe preflight status that are rejected until replaced. `pnpm release:live-plan` is the safe handoff command for the remaining external work: it reads `roadmap:completion-audit`, groups missing checks by runtime, release evidence, GA sign-off, and final acceptance, and prints only artifact names, command templates, env-key names, and blocking roadmap/PRD row references. For the release-evidence action group, those artifact names include both the final `release-evidence.json` ledger and prerequisite live proof inputs: `live-secrets-probe.json`, `live-canary-results.json`, `live-provider-submit-proof.json`, `live-calendar-smoke.json`, `live-dispatch-proof.json`, and `live-7-day-soak.json`; parseable input-shape examples for those files live under `docs/examples/` with `.example.json` suffixes and are blocked as non-live examples. `pnpm release:live-inputs:validate` validates those six prerequisite inputs together before any append step, checks that every generated release-evidence record passes the shared evaluator, and rejects example markers. `pnpm roadmap:live-acceptance` is the one-command final bundle gate: it validates runtime config, prerequisite live inputs when `LIVE_ACCEPTANCE_VALIDATE_INPUT_BUNDLE=true` and `LIVE_PROOF_INPUTS_ASSERT_LIVE=true`, the release ledger, GA sign-off, acceptance package, and `roadmap:completion-audit`, then exits non-zero if any layer is still local-only, stale, mismatched, or missing. `pnpm runtime:preflight` checks the production runtime envelope and, by default, probes Postgres via `DATABASE_URL`, Redis via `REDIS_URL`, Telegram `getMe`, the configured LLM endpoint, S3-compatible object storage with a temporary write/read/delete roundtrip, and the local encrypted secret store when that backend is selected; it also requires `QUEUE_BACKEND=bullmq` and `TELEGRAM_WEBHOOK_SECRET` when a live bot token is configured. It writes only hashes and booleans for sensitive config, including the webhook secret. Production irreversible actions also require `OBJECT_STORAGE_BACKEND=s3_compatible` so proof artifacts are written to durable S3-compatible storage rather than local filesystem. The final audit requires that preflight report to match the current runtime env and be fresh within 24 hours by default; set `RUNTIME_PREFLIGHT_MAX_AGE_HOURS` on `pnpm roadmap:completion-audit` to change that window. `pnpm llm:smoke` is the narrower LLM-only diagnostic: it performs one schema-validated JSON call against the configured OpenAI-compatible endpoint, records model/prompt/input hashes and latency, and never emits API keys, raw prompts, or endpoint URLs. `pnpm secrets:probe` can upsert a verified 24-hour `external_secrets_backend` evidence record for `local_encrypted_file`; it hashes the store root and never emits the master key. The same probe inventories safe `SecretReference` handles and upserts 24-hour `live_credentials_configured` only when every expected provider and the Telegram bot credential are covered; set `SECRET_STORE_PROBE_REQUIRE_CREDENTIALS=true` to make incomplete inventory fail the command or `SECRET_STORE_PROBE_EVIDENCE_TTL_HOURS` to change the TTL. `pnpm secrets:evidence` converts externally captured managed secret-store probe output for `vault`, `aws_secrets_manager`, `gcp_secret_manager`, or `local_encrypted_file` into expiring `external_secrets_backend` evidence, rejects raw secret-like input fields, hashes backend scope values, and can add `live_credentials_configured` from safe secret-reference handles. `pnpm canary:evidence` converts externally captured live canary results into `live_canary_passed` records only when the artifact has per-provider `canaryRunId`/`checkedAt` data, a non-fixture source, and `CANARY_EVIDENCE_ASSERT_LIVE=true`; generated canary evidence expires after 24 hours by default, configurable with `CANARY_EVIDENCE_TTL_HOURS`. `pnpm canary:live-smoke` performs read-only live canaries directly, using HTTP GET/status/text checks for provider targets and Telegram `getMe` for the Telegram target; it emits hashes for URLs and text anchors, never response bodies or tokens, and can append `live_canary_passed`. `pnpm release:evidence:validate` and `pnpm acceptance:package` derive provider readiness from those current canary records and explicit provider configs instead of accepting static selector metadata as live proof; in production missing canary, replay, or provider config inputs stay blocked as `canary_missing`, `replay_missing`, or `disable_switch_missing`, and both reports fail closed when the runtime envelope is not production-shaped. `pnpm provider-submit:evidence` converts externally captured first-provider submit proof into expiring `provider_submit_proof_ready` only for provider transport, `send_application`, submitted status, hashed idempotency, approved draft hash, submitted timestamp, and a live source; generated submit evidence expires 24 hours after submission by default, configurable with `PROVIDER_SUBMIT_EVIDENCE_TTL_HOURS`, and rejects raw application payload fields. Runtime approved-submit regenerates the approved draft payload and rechecks its content hash before calling any live submit executor. `pnpm calendar:evidence` converts an external calendar smoke artifact into `calendar_integration_ready` only when live read, conflict, and write checks all passed; generated calendar readiness evidence expires after 24 hours by default, configurable with `CALENDAR_EVIDENCE_TTL_HOURS`. `pnpm google-calendar:smoke` executes those checks directly against Google Calendar when `GOOGLE_CALENDAR_SMOKE_CONFIRM_LIVE=true`: it runs freeBusy, creates a temporary opaque event, verifies conflict detection, deletes the event, emits only hashes for identifiers, and can append `calendar_integration_ready`. `pnpm outbound:evidence` converts a live sent dispatch proof into expiring `outbound_dispatch_proof_ready` only for `provider`, `telegram`, or `calendar` transport with sent status, hashed idempotency/text values, `deliveredAt`, and a future `expiresAt`; generated dispatch evidence expires 24 hours after delivery by default, configurable with `OUTBOUND_EVIDENCE_TTL_HOURS`, and rejects raw message fields. `pnpm telegram:dispatch-smoke` performs the Telegram send path itself when `TELEGRAM_DISPATCH_CONFIRM_LIVE=true`, emits only hashes plus Telegram message-id hash, and can append expiring `outbound_dispatch_proof_ready` without storing raw message text. `pnpm soak:evidence` converts a real 7-day soak artifact into `seven_day_soak_passed` only when `startedAt`/`completedAt` prove at least seven elapsed days and the duplicate, proof coverage, state-loss, unsupported-fact, and drill counters satisfy the gate; generated soak evidence expires 30 days after `completedAt` by default, configurable with `SOAK_EVIDENCE_TTL_DAYS`. `pnpm ga-signoff:validate` checks the sign-off file without running the full acceptance package. `pnpm release:evidence:validate` checks the release evidence ledger and release gates without running the fixture soak. `pnpm acceptance:package` is fail-closed and exits non-zero whenever evidence, runtime, release gates, or GA sign-off do not satisfy acceptance.

Runtime approved-submit can also be backed by an audited external submit executor by setting the provider config entry to `runtimeKind: "live"` with `liveSubmitEndpoint` and optional `liveSubmitAuthTokenEnv`. The wrapper stores only proof metadata in-process and still requires the approval, policy, canary, release evidence, and release gates before a submit can be treated as production-ready.

## Evidence Ledger

Use `GET /release-evidence` to inspect accepted evidence and summary blockers.

Use `POST /release-evidence` to record dated evidence:

- `live_credentials_configured`
- `external_secrets_backend`
- `live_canary_passed`
- `provider_submit_proof_ready`
- `calendar_integration_ready`
- `seven_day_soak_passed`
- `outbound_dispatch_proof_ready`

Evidence must include `source`, `status`, and type-specific metadata. `POST /release-evidence` rejects new `passed` evidence with missing metadata, expired `expiresAt`, raw secret-like values, template markers, or fixture-only dispatch proof. Imported or historical weak evidence is ignored by summaries and reported as `invalid_release_evidence:*`.

| Evidence type | Required metadata |
|---|---|
| `live_credentials_configured` | `secretReferenceIds`, `coveredProviderIds`, `telegramBot: true`, `checkedAt`, future `expiresAt` |
| `external_secrets_backend` | `backend` set to `vault`, `aws_secrets_manager`, `gcp_secret_manager`, or `local_encrypted_file`, plus `accessCheck: true` or `probe: "passed"`, `checkedAt`, future `expiresAt` |
| `live_canary_passed` | non-null `providerId`, `canaryRunId`, `checkedAt`, future `expiresAt`, and `result: "passed"` or `status: "passed"` |
| `provider_submit_proof_ready` | non-null provider `providerId`, `applicationId`, `proofId`, `action: "send_application"`, `transport: "provider"`, `idempotencyKeyHash`, `draftHash`, `submittedAt`, future `expiresAt`, and `submitStatus: "submitted"` or `proofStatus: "submitted"` |
| `calendar_integration_ready` | `calendarProvider`, `checkedAt`, future `expiresAt`, `readCheck: true`, `conflictCheck: true`, `writeCheck: true` |
| `seven_day_soak_passed` | at least 7 elapsed days proven by `startedAt`/`completedAt`, future `expiresAt`, `duplicateApplicationCount: 0`, `proofCoveragePercent: 100`, `stateLossDetected: false`, `unsupportedFactCount: 0`, `incidentDrillPassed: true`, `rollbackDrillPassed: true` |
| `outbound_dispatch_proof_ready` | `proofId`, live `transport` (`provider`, `telegram`, or `calendar`), `idempotencyKeyHash`, `textHash`, `deliveredAt`, future `expiresAt`, and `deliveryStatus: "sent"` or `proofStatus: "sent"` |
