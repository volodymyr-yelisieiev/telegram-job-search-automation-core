Status meanings:

- `Implemented/tested`: repository contains working local-safe code, tests, and docs.
- `Implemented/local-safe`: production-shaped code or documentation exists, but live external systems remain disabled.
- `External evidence required`: cannot be completed honestly without provider credentials, live accounts, calendar/secrets infrastructure, staged rollout, or a dated soak.

## Self-Check

This matrix records repository evidence for all 40 roadmap sprints, including accepted ADRs, provider playbooks, incident runbooks, an executable accelerated soak, final soak report artifact, acceptance package generator, a final documentation-pack manifest, a compact local roadmap gate wired into `pnpm verify`, a redacted live-completion handoff plan, a parseable `docs/examples/release-evidence.example.json` template, a parseable `docs/examples/ga-signoff.example.json` template, a parseable schema-only `docs/examples/runtime-preflight.example.json` template, and parseable prerequisite live-proof input examples for secrets, canaries, provider submit proof, calendar smoke, outbound dispatch proof, and 7-day soak. `pnpm release:live-inputs:validate` checks those six prerequisite inputs together before any ledger mutation, runs their generated records through the shared release-evidence evaluator, and rejects example markers. `pnpm roadmap:live-acceptance` is the single fail-closed live bundle gate over runtime config, optional prerequisite live inputs, release-evidence validation, GA sign-off validation, acceptance package, and final roadmap completion audit. The templates are intentionally blocked from satisfying live gates until example markers are replaced and a fresh production runtime preflight is generated. Queueing includes typed in-memory and BullMQ payload validation, BullMQ job envelopes, worker-side task reconstruction, task-run lifecycle persistence, retryable failure handoff to BullMQ, and DLQ routing for no-retry failures. Release evidence is type-specific and rejects missing metadata, raw secret-like values, fixture/local-only or untimestamped dispatch proof, weak soak claims, non-expiring credentials/secrets/canary/provider-submit/calendar/soak/outbound claims, and template markers both at API write time and summary evaluation time. Provider readiness inside release/acceptance reports is derived from current accepted canary evidence and explicit provider config disable-switch coverage, so static selector metadata alone cannot satisfy live readiness; production reports keep missing canary, replay, or provider config inputs blocked as `canary_missing`, `replay_missing`, or `disable_switch_missing`. Production review-first/controlled submit readiness also rejects fixture-backed auto-apply providers until a live-capable provider module is installed. Runtime preflight now also requires `STATE_BACKEND=postgres` with `DATABASE_URL`, `QUEUE_BACKEND=bullmq` with `REDIS_URL`, a configured live-submit endpoint plus token env for irreversible production submit modes, and release/acceptance CLI reports fail closed when the runtime envelope is not production-shaped. Approved submit regenerates the draft payload and rechecks its content hash before calling any live submit executor. GA sign-off requires explicit issue-register, runbook-drill, residual-risk, and maintenance-plan evidence references, and example references remain blocked. Proof artifacts have in-memory, filesystem, and S3-compatible storage adapters with cleanup/delete support; production irreversible config rejects filesystem-only proof storage, and runtime preflight performs a temporary S3-compatible write/read/delete probe before final audit can pass. Live Telegram runtime preflight also requires a configured guarded webhook secret and stores only a readiness boolean/hash. The remaining unresolved items are explicitly external: live credentials/accounts, live provider canaries, real submit/reply/calendar smoke execution, external secrets-manager provisioning, a dated 7-day soak, and real product/engineering/operations/security sign-off. Local-safe code keeps all irreversible behavior disabled until those gates are satisfied.

`pnpm roadmap:compliance` verifies that the source roadmap has exactly sprint IDs 0-39, that every sprint has a matrix row with an allowed status and non-empty evidence, and that the delivery roadmap board uses the matching status taxonomy.

`pnpm roadmap:local-gates` is included in `pnpm verify` and CI. It requires roadmap compliance, a valid final documentation pack, queue resilience checks for duplicate suppression, DLQ visibility, retry enqueue, and worker-restart recovery, and a generated redacted live handoff plan with the expected action groups and blocking roadmap/PRD row references. It does not satisfy or fake live evidence.

`pnpm roadmap:live-acceptance` runs the final live bundle as one fail-closed check and can also require `LIVE_ACCEPTANCE_VALIDATE_INPUT_BUNDLE=true` plus `LIVE_PROOF_INPUTS_ASSERT_LIVE=true` to revalidate the six prerequisite live input files before accepting the already-built ledger. `pnpm roadmap:completion-audit` is the stricter final-goal check. It intentionally fails while local-safe/external-only sprint or PRD rows lack their required live checks, while `release-evidence.json`, `ga-signoff.json`, or `runtime-preflight.json` are missing, invalid, expired, mismatched with the current runtime env, or missing required preflight checks/probes, while aggregate release-gate/acceptance-package checks fail, or while the runtime is not production-shaped. The runtime preflight freshness window defaults to 24 hours and can be overridden with `RUNTIME_PREFLIGHT_MAX_AGE_HOURS`. The audit emits `requiredLiveChecks` and `missingLiveChecks` for each unresolved row. Passing `roadmap:compliance` alone is not sufficient to claim GA completion.
